# Feature Specification: تحسين النسخ الاحتياطي — جداول شاملة + تصدير CSV + استعادة بتاريخ

**Feature Branch**: `016-backup-restore-csv`
**Created**: 2026-05-21
**Status**: Draft

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 — تشمل النسخة كل الجداول التشغيلية (Priority: P1)

المسؤول يضغط "تشغيل الآن" فتشمل النسخة الاحتياطية **جميع** جداول البيانات التشغيلية (الموظفين، الشركات، المشاريع، المسيّرات، المستخلصات، المستخدمين، الإعدادات، التنبيهات المقروءة، أسعار المسميات الوظيفية) — لا تُفقَد أي بيانات عند الاستعادة.

**Why this priority**: الهدف الأساسي من النسخة هو حماية **كل** البيانات. نسخة ناقصة = ضمان ناقص.

**Independent Test**: اضغط "تشغيل الآن" ← حمّل الملف ← افتح JSON ← تحقق أن المفاتيح تشمل: `users, system_settings, read_alerts, project_job_title_rates, extract_invoices, extract_invoice_lines` بالإضافة للـ 12 الحاليين.

**Acceptance Scenarios**:

1. **Given** المستخدم مسؤول (admin)، **When** يضغط "تشغيل الآن"، **Then** تُنشأ نسخة تحتوي على ≥18 جدول تشغيلي موثَّق في metadata.
2. **Given** النسخة مكتملة، **When** يفتح الملف، **Then** يجد حقلاً `table_record_counts` يوضح عدد السجلات لكل جدول.
3. **Given** محاولة من مستخدم غير admin، **When** يضغط أي زر في صفحة النسخ، **Then** يُرفَض الطلب بدون تنفيذ.

---

### User Story 2 — استعادة النظام لنسخة سابقة (Priority: P1)

المسؤول يختار نسخة احتياطية من السجل بتاريخ محدد، يراجع ملخص البيانات، يؤكد مرتين، فيستعيد النظام لتلك الحالة — مع حفظ snapshot تلقائي للوضع الراهن أولاً.

**Why this priority**: سبب وجود النسخ الاحتياطية أصلاً. بدون استعادة، النسخة بلا قيمة عملية.

**Independent Test**: أنشئ نسخة ← أدخل بيانات جديدة ← استعد من النسخة السابقة ← تحقق أن البيانات الجديدة اختفت والقديمة عادت ← تحقق أن snapshot "ما قبل الاستعادة" ظهر في السجل.

**Acceptance Scenarios**:

1. **Given** قائمة النسخ تعرض نسخة، **When** يضغط "معاينة واستعادة"، **Then** تظهر شاشة تعرض: تاريخ النسخة، عدد الجداول، إجمالي السجلات لكل جدول، حجم الملف.
2. **Given** شاشة المعاينة، **When** يضغط "استعادة"، **Then** يظهر تحذير صريح + خانة نصية لكتابة كلمة تأكيد قبل تفعيل زر التنفيذ.
3. **Given** التأكيد المزدوج، **When** يُنفَّذ الطلب، **Then** يُنشأ أولاً snapshot من نوع `pre-restore-snapshot` ← ثم تُستعاد البيانات — إذا فشل الـ snapshot تُلغى العملية كلياً.
4. **Given** اكتمال الاستعادة، **When** يعود للنظام، **Then** يجد البيانات مطابقة لتاريخ النسخة المختارة، ويرى السجل يتضمن سجل عملية الاستعادة.
5. **Given** فشل الاستعادة في منتصفها، **Then** لا يتغير أي شيء (atomic rollback) ويظهر تنبيه بوجود snapshot يمكن الرجوع إليه.

### User Story 3 — تنزيل CSV لكل الجداول (Priority: P2)

المسؤول يضغط "تحميل CSV" على نسخة محددة فيحصل على ZIP يحتوي ملف CSV واحد لكل جدول — قابل للفتح مباشرة في Excel مع دعم كامل للنص العربي.

**Why this priority**: يُمكّن مراجعة البيانات بدون أدوات تقنية، مفيد للمحاسبين والمراجعين.

**Independent Test**: حمّل CSV ZIP ← افتح أي ملف CSV في Excel ← تحقق من ظهور الحروف العربية بدون تشويه.

**Acceptance Scenarios**:

1. **Given** نسخة مكتملة في السجل، **When** يضغط "تحميل CSV"، **Then** يبدأ تحميل ZIP يحمل اسم `backup-[تاريخ].zip`.
2. **Given** ملف ZIP، **When** يفتحه، **Then** يجد ملف CSV مستقل لكل جدول مُسمَّى باسمه.
3. **Given** ملف CSV، **When** يفتحه في Excel، **Then** تظهر الأعمدة بعناوين واضحة والقيم العربية سليمة (UTF-8 BOM).
4. **Given** جدول يحتوي حقل JSON أو نص متعدد الأسطر، **Then** يظهر في CSV داخل علامات اقتباس مع escape سليم.

---

### Edge Cases

- نسخة ملفها محذوف من Storage لكن سجلها لا يزال في القاعدة → زر التنزيل/الاستعادة يُظهر خطأ واضح بدل السقوط.
- بنية جدول تغيّرت بين تاريخ النسخة والاستعادة (أعمدة جديدة/محذوفة) → الاستعادة تفشل بأمان مع رسالة توضح التعارض.
- محاولة استعادة أثناء تشغيل نسخة احتياطية أخرى → رفض مع رسالة "عملية جارية".
- snapshot الوقائي فشل (مساحة ممتلئة) → الاستعادة تُلغى كلياً، لا بيانات تُفقَد.
- انقطاع اتصال أثناء الاستعادة → atomic rollback، حالة الاستعادة `failed` في السجل.
- المستخدم المُنفِّذ موجود في النسخة بصلاحيات مختلفة → بعد الاستعادة يستمر جلسته الحالية، لكن صلاحياته تتغير عند تسجيل الدخول التالي.
- حجم النسخة كبير جداً يتجاوز حدود Edge Function → إخفاق مع رسالة توجيهية.

## Requirements *(mandatory)*

### Functional Requirements

**النسخ الاحتياطي:**
- **FR-001**: النسخة تشمل جميع الجداول التشغيلية في `public` schema — على الأقل: `employees, companies, projects, transfer_procedures, employee_obligation_headers, employee_obligation_lines, payroll_runs, payroll_entries, payroll_entry_components, payroll_slips, notifications, saved_searches, users, system_settings, read_alerts, project_job_title_rates, extract_invoices, extract_invoice_lines`.
- **FR-002**: تُستثنى صراحةً جداول: `auth.users` (passwords/sessions — قيود المنصة)، `backup_history`، `restore_history`، جداول الـ audit/log.
- **FR-003**: metadata كل نسخة تتضمن `table_record_counts` (اسم الجدول → عدد السجلات) مخزَّناً في `backup_history.tables_included` أو حقل منفصل.
- **FR-004**: تحذير ثابت في الواجهة يوضح أن بيانات كلمات المرور وجلسات الدخول خارج نطاق النسخة الاحتياطية.

**التنزيل:**
- **FR-005**: زر "تحميل JSON" يوفر رابط موقَّع (signed URL) للملف `.json.gz` — صلاحيته 1 ساعة.
- **FR-006**: زر "تحميل CSV" يُولِّد ملف ZIP يحتوي ملف CSV لكل جدول — يُنتَج من ملف JSON الأصلي (لا تخزين مزدوج).
- **FR-007**: ملفات CSV بترميز **UTF-8 with BOM** وfأميد سليم للحقول متعددة الأسطر والحقول المحتوية فواصل.

**الاستعادة:**
- **FR-008**: كل نسخة مكتملة في السجل تعرض زر "معاينة واستعادة".
- **FR-009**: شاشة المعاينة تعرض: تاريخ النسخة، النوع، المنشئ، الجداول المشمولة وعدد سجلاتها، الحجم الإجمالي.
- **FR-010**: التأكيد المزدوج إلزامي: (1) زر "بدء الاستعادة" + تحذير ← (2) حقل نصي يكتب فيه "استعادة" + زر "تأكيد نهائي".
- **FR-011**: قبل تنفيذ الاستعادة، يُنشأ تلقائياً snapshot من نوع `pre-restore-snapshot` — إذا فشل يُلغى كل شيء.
- **FR-012**: الاستعادة تُنفَّذ على الخادم (Edge Function) بصلاحيات إدارية — العميل لا يُجري أي عمليات DB مباشرة.
- **FR-013**: الاستعادة atomic — تنجح كلياً أو تُلغى كلياً (rollback عند أي خطأ).
- **FR-014**: ترتيب المسح والإدخال يحترم تبعيات الجداول (FK constraints).
- **FR-015**: كل عملية استعادة تُسجَّل في `restore_history` (المُنفِّذ، النسخة المصدر، الحالة، الوقت، snapshot المرتبط).
- **FR-016**: بعد الاستعادة، الواجهة تُحدَّث وتُظهر إشعار نجاح أو فشل مع رابط للـ snapshot الوقائي.

**الصلاحيات:**
- **FR-017**: كل عمليات النسخ والاستعادة والتنزيل للـ admin فقط — غير الـ admin لا يرى الأزرار، وأي طلب API يُرفَض (403).
- **FR-018**: كل عملية استعادة تُسجَّل في audit log.

### Key Entities

| الكيان | الوصف |
|---|---|
| **Backup Record** | سجل في `backup_history`: ID، تاريخ، منشئ، نوع (manual/scheduled/pre-restore-snapshot)، مسار الملف، الحجم، `table_record_counts` (JSON)، الحالة |
| **Backup Archive** | الملف الفعلي: JSON مضغوط يحتوي بيانات الجداول + metadata |
| **Restore Operation** | سجل في `restore_history`: ID، backup_id المصدر، المُنفِّذ، الوقت، الحالة، snapshot_id الوقائي، رسالة النتيجة |
| **Pre-Restore Snapshot** | Backup Record من نوع `pre-restore-snapshot` — يُنشأ تلقائياً قبل كل استعادة |
| **Restore Preview** | كيان مؤقت (غير مخزَّن): يُنتَج من metadata النسخة لعرض الملخص قبل التأكيد |

## Success Criteria *(mandatory)*

- **SC-001**: النسخة تشمل 100% من الجداول التشغيلية المُعرَّفة في FR-001 — يمكن التحقق بفتح الملف.
- **SC-002**: ملفات CSV تعرض النص العربي بشكل صحيح عند الفتح في Microsoft Excel بدون أي إعداد إضافي.
- **SC-003**: زمن إنشاء النسخة لا يتجاوز 60 ثانية لقاعدة بيانات بحجم 100,000 سجل إجمالي.
- **SC-004**: مستحيل تشغيل الاستعادة بأقل من خطوتي تأكيد (قابل للتحقق من الواجهة).
- **SC-005**: لا تُنفَّذ أي استعادة بدون snapshot وقائي مُنشأ أولاً (صفر استعادات بلا snapshot).
- **SC-006**: 100% من عمليات الاستعادة مُسجَّلة في `restore_history` بكامل التفاصيل.
- **SC-007**: مستخدم غير admin لا يستطيع تنفيذ أي عملية نسخ أو استعادة — الواجهة والـ API كلاهما يمنعان.
- **SC-008**: فشل الاستعادة في أي خطوة لا يُخلِّف بيانات جزئية (atomic behavior).

## Assumptions

- **A-001**: Schema الجداول ثابت بين تاريخ النسخة والاستعادة — عدم التطابق يُفشل الاستعادة بأمان (لا استعادة جزئية).
- **A-002**: حساب Supabase Pro — Edge Functions تدعم timeout كافياً لعمليات الاستعادة (< 5 دقائق).
- **A-003**: مساحة Storage كافية لاحتفاظ معقول بالنسخ + snapshots الوقائية.
- **A-004**: الاستعادة الجزئية (جدول واحد) خارج نطاق هذا الـ spec — استعادة كاملة فقط.
- **A-005**: وضع الصيانة أثناء الاستعادة مقبول (دقائق قليلة، نادر الحدوث).
- **A-006**: بعد الاستعادة، المستخدمون يُسجّلون دخول طبيعي بكلمات مرورهم الحالية — `auth.users` لا يتأثر.
- **A-007**: "استعادة بتاريخ" = اختيار من نسخ مخزَّنة فعلياً — ليس point-in-time recovery مستمر (ذاك يتطلب Supabase PITR مباشرة).
