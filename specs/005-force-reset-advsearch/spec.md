- Q: ما سلوك "إعادة تعيين كلمة المرور" بالضبط؟ → A: الأدمن يكتب كلمة مرور جديدة (≥8 أحرف) في Dialog

- Q: هل يُمنع الأدمن من إعادة تعيين كلمة مروره الخاصة؟ → A: نعم، الزر معطّل/مخفي للمستخدم المسجّل حالياً

## User Scenarios & Testing *(mandatory)*

### User Story 1 - إعادة تعيين كلمة مرور مستخدم (Priority: P1)

الأدمن يحتاج لإعادة تعيين كلمة مرور مستخدم (مثلاً: نسي كلمته أو تم اختراق حسابه). يذهب لصفحة الإعدادات ← تبويب المستخدمون والصلاحيات، يجد زر "إعادة تعيين" في صف المستخدم المطلوب، يفتح Dialog يدخل فيه الكلمة الجديدة ويؤكدها، ثم يحفظ.

**Why this priority**: وظيفة إدارية حرجة — بدونها الأدمن لا يستطيع مساعدة مستخدم فقد وصوله.

**Independent Test**: يمكن اختباره بالكامل بتسجيل دخول كأدمن، فتح صفحة الإعدادات، وتعيين كلمة مرور جديدة لمستخدم آخر، ثم التحقق من أن المستخدم يقدر يسجّل دخوله بالكلمة الجديدة.

**Acceptance Scenarios**:

1. **Given** الأدمن في تبويب "المستخدمون والصلاحيات"، **When** يضغط زر "إعادة تعيين" في صف مستخدم آخر، **Then** يفتح Dialog يحتوي حقل كلمة المرور الجديدة وحقل التأكيد
2. **Given** الأدمن أدخل كلمة مرور جديدة (≥8 أحرف) وأكّدها، **When** يضغط "حفظ"، **Then** تُحدَّث كلمة المرور ويظهر toast نجاح وينغلق الـ Dialog
3. **Given** الأدمن أدخل كلمتين مرور غير متطابقتين، **When** يضغط "حفظ"، **Then** يظهر خطأ تحقق "كلمتا المرور غير متطابقتين" ولا تُرسَل الطلب
4. **Given** الأدمن أدخل كلمة مرور أقل من 8 أحرف، **When** يحاول الحفظ، **Then** يظهر خطأ تحقق "كلمة المرور يجب أن تكون 8 أحرف على الأقل"
5. **Given** الأدمن في تبويب المستخدمون والصلاحيات، **When** ينظر لصفه الخاص، **Then** زر "إعادة تعيين" غير موجود أو معطّل (حماية ذاتية)

---

### User Story 2 - ريفاكتور البحث المتقدم: استخراج مكون الفلاتر النشطة (Priority: P2)

مكون `<ActiveFilterChips />` مستقل يعرض شرائح الفلاتر النشطة (الجنسية، المهنة، المؤسسة، إلخ) مع زر حذف لكل منها. مستخرج من `AdvancedSearch.tsx` لتقليل حجمه وتحسين قابلية الصيانة.

**Why this priority**: دين تقني — 174 سطر JSX مضمّن في الصفحة الرئيسية لا يخدم أي غرض إضافي وراء الاستخراج.

**Independent Test**: يمكن اختباره بفتح صفحة البحث المتقدم، تطبيق أي فلتر، والتحقق من ظهور شريحة الفلتر مع إمكانية إلغائه بالضغط على X.

**Acceptance Scenarios**:

1. **Given** مستخدم طبّق فلتراً في البحث المتقدم، **When** تعرض الصفحة، **Then** تظهر شرائح الفلاتر النشطة بنفس الشكل والوظيفة السابقين تماماً
2. **Given** مستخدم يضغط X على شريحة فلتر، **When** تُعالَج النقرة، **Then** يُلغى الفلتر وتختفي الشريحة فوراً
3. **Given** لا توجد فلاتر نشطة، **When** تعرض الصفحة، **Then** لا تظهر أي شرائح (سلوك محافَظ عليه)

---

### User Story 3 - ريفاكتور البحث المتقدم: حل prop drilling في FiltersModal (Priority: P3)

`AdvancedSearchFiltersModal` يستقبل كائن `search` واحداً بدل 20+ prop فردي — تقليل verbosity وتسهيل إضافة فلاتر مستقبلاً.

**Why this priority**: تحسين جودة كود — الوظيفة لا تتغير، لكن المحافظة على الكود تصبح أسهل.

**Independent Test**: جميع وظائف الفلاتر الموجودة تعمل بنفس الطريقة بعد الريفاكتور — لا regression.

**Acceptance Scenarios**:

1. **Given** المستخدم فتح modal الفلاتر، **When** يغيّر أي فلتر، **Then** تُحدَّث النتائج فوراً بنفس السلوك السابق
2. **Given** المستخدم يحفظ بحثاً أو يحمّل بحثاً محفوظاً، **When** يُنفَّذ الإجراء، **Then** يعمل بدون أخطاء

---

### Edge Cases

- ماذا يحدث لو فشل الـ API أثناء تعيين كلمة المرور؟ → يُعرض toast خطأ، يبقى الـ Dialog مفتوحاً، لا تتغير كلمة المرور
- ماذا لو كان المستخدم المستهدف غير موجود (حُذف للتو)؟ → الـ API يُعيد خطأ 404، يُعرض للأدمن "المستخدم غير موجود"
- هل يمكن للأدمن تعيين نفس كلمة المرور الحالية؟ → نعم، لا قيد على ذلك (Supabase لا يمنعه)
- لو الأدمن أغلق الـ Dialog قبل الحفظ؟ → لا شيء يتغير، الـ Dialog ينغلق فقط

---

## Requirements *(mandatory)*

### Functional Requirements

**الهدف الأول — Force Password Reset:**

- **FR-001**: النظام يجب أن يعرض زر "إعادة تعيين كلمة المرور" في عمود الإجراءات لكل مستخدم في جدول "المستخدمون والصلاحيات"
- **FR-002**: الزر يجب أن يكون غائباً أو معطّلاً للمستخدم المسجّل دخوله حالياً (الأدمن نفسه)
- **FR-003**: عند الضغط على الزر، يفتح Dialog يحتوي: حقل كلمة المرور الجديدة + حقل تأكيدها + زر "حفظ" + زر "إلغاء"
- **FR-004**: النظام يجب أن يتحقق من: كلمة المرور ≥8 أحرف، وتطابق حقل التأكيد، قبل إرسال الطلب
- **FR-005**: عند نجاح العملية، يُغلق الـ Dialog ويظهر toast نجاح
- **FR-006**: عند فشل العملية، يُعرض toast خطأ ويبقى الـ Dialog مفتوحاً مع الحقول كما هي

**الهدف الثاني — AdvancedSearch Refactor:**

- **FR-007**: يُستخرج مكون `<ActiveFilterChips />` مستقل يستقبل كائن `search` ويعرض شرائح الفلاتر النشطة مع وظيفة الحذف
- **FR-008**: `AdvancedSearchFiltersModal` يُعاد هيكلته ليستقبل كائن `search` كاملاً بدل 20+ prop فردي
- **FR-009**: جميع وظائف البحث والفلترة والحفظ والتصدير يجب أن تعمل بدون أي تغيير في السلوك بعد الريفاكتور

### Key Entities

- **User**: المستخدم المُدار (id, email, full_name, role, is_active) — القيد: لا يمكن الأدمن إعادة تعيين باسورد نفسه
- **PasswordResetForm**: بيانات النموذج (newPassword: string ≥8, confirmPassword: string) — تحقق على مستوى الـ UI فقط
- **SearchState (search object)**: كائن يضم كامل حالة البحث المتقدم (filters, pagination, modal states, handlers) — يُمرَّر كـ prop واحد للمكونات الفرعية

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: الأدمن يقدر يعيد تعيين كلمة مرور أي مستخدم في أقل من 30 ثانية من لحظة قرار التعيين
- **SC-002**: لا يوجد أي حالة تتيح للأدمن تعيين كلمة مرور أقل من 8 أحرف أو غير مؤكدة لأي مستخدم
- **SC-003**: الأدمن لا يقدر أبداً أن يعيد تعيين كلمة مرور حسابه الخاص عبر هذه الواجهة
- **SC-004**: جميع وظائف البحث المتقدم تعمل بنفس الطريقة بعد الريفاكتور — صفر regression مُبلَّغ عنه
- **SC-005**: حجم `AdvancedSearch.tsx` ينخفض بما لا يقل عن 150 سطراً بعد استخراج `<ActiveFilterChips />`
- **SC-006**: `AdvancedSearchFiltersModal` لا يستقبل أكثر من 5 props بعد الريفاكتور

---

## Assumptions

- النظام داخلي بالكامل — المستخدمون موظفون أو مدراء، لا يوجد self-service لتعيين كلمة المرور
- يوجد أدمن واحد فعّال في النظام في الغالب — منع الأدمن من تعيين باسورد نفسه عبر هذه الواجهة كافٍ للحماية
- الـ Express API server (`PATCH /api/admin/users/:id`) يعمل وجاهز — لا يحتاج تعديل في الـ backend
- الـ `UpdateAdminUserBody` schema في `openapi.yaml` يدعم حقل `password` بالفعل — لا تعديل في الـ API spec
- `AdvancedSearchFiltersModal` يستقبل كائن `search` بنفس واجهة `useAdvancedSearchFilters` — أي تغيير في الهوك يستلزم تحديث الـ props
- الريفاكتور لا يغيّر أي سلوك مرئي — اختبار يدوي كافٍ للتحقق من غياب الـ regression
