# Feature Specification: إصلاح تهنيج تبويب تصدير الموظفين

**Feature Branch**: `011-fix-export-performance`  
**Created**: 2026-05-18  
**Status**: Draft  

## User Scenarios & Testing *(mandatory)*

### User Story 1 - التبديل السلس بين نوعي التصدير (Priority: P1)

المستخدم يفتح تبويب الاستيراد/التصدير، ويختار "تصدير موظفين"، ثم يغير نوع التصدير من "تصدير أساسي" إلى "تصدير شهري بالتفاصيل المالية" (أو العكس) — المتصفح يستجيب فوراً بدون تعليق أو تجميد.

**Why this priority**: التهنيج الحالي يستغرق 2+ ثانية ويعطل المتصفح — هذا يمنع الاستخدام الطبيعي للميزة.

**Independent Test**: افتح تبويب التصدير مع 100+ موظف → غيّر نوع التصدير → القائمة تظل مستجيبة والمتصفح لا يتجمد.

**Acceptance Scenarios**:

1. **Given** قائمة موظفين بها 100+ موظف مُحمَّلة في تبويب التصدير، **When** المستخدم يغير نوع التصدير من "أساسي" إلى "شهري"، **Then** الواجهة تستجيب في أقل من 200 ملي ثانية بدون تجمد مرئي للمتصفح.

2. **Given** تم تحديد موظفين من القائمة، **When** المستخدم يغير نوع التصدير، **Then** تظل التحديدات محفوظة والقائمة تعرض نفس الموظفين المصفاة.

3. **Given** فلاتر نشطة (تصفية بالشركة أو المشروع)، **When** المستخدم يغير نوع التصدير، **Then** تظل الفلاتر محفوظة والأداء سريع.

---

### User Story 2 - التفاعل السلس عند التحديد والإلغاء (Priority: P2)

المستخدم يحدد أو يلغي تحديد موظفين فرديين أو الكل — كل نقرة تُحدِّث الواجهة فوراً بدون تأخير.

**Why this priority**: تحديد الموظفين يستدعي re-render للقائمة كلها، مما يسبب تأخيراً مع الأعداد الكبيرة.

**Independent Test**: حدد 50 موظفاً واحداً تلو الآخر → كل نقرة تُستجاب خلال 100ms.

**Acceptance Scenarios**:

1. **Given** قائمة 200 موظف، **When** المستخدم ينقر على موظف لتحديده، **Then** تُحدَّث أيقونة التحديد فوراً دون تأخير ملحوظ.

2. **Given** عدة موظفين محددين، **When** المستخدم ينقر "تحديد الكل"، **Then** تُحدَّث جميع الصفوف دفعة واحدة سريعاً.

---

### User Story 3 - أداء مستقر مع تغيير الفلاتر (Priority: P3)

عند تغيير فلاتر البحث أو الشركة أو المشروع، تُعاد قائمة الموظفين المصفاة بسرعة بدون إعادة حساب البيانات الثابتة.

**Why this priority**: تحسين الأداء العام للصفحة يشمل جميع تفاعلات المستخدم.

**Independent Test**: غيّر نص البحث تكراراً → القائمة تتصفى سريعاً بدون تجمد.

**Acceptance Scenarios**:

1. **Given** قائمة 200 موظف، **When** المستخدم يكتب في حقل البحث، **Then** تتحدث القائمة خلال أقل من 300ms لكل حرف.

---

### Edge Cases

- ماذا يحدث إذا كان عدد الموظفين صفراً؟ → القائمة فارغة بدون أخطاء.
- ماذا إذا تغيرت بيانات الموظفين أثناء استخدام الصفحة؟ → البيانات المُحمَّلة تظل كما هي حتى إعادة التحميل اليدوي.
- ماذا على الموبايل مع قائمة كبيرة؟ → نفس معايير الأداء تنطبق على عرض الكروت.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: الواجهة MUST تستجيب لتغيير نوع التصدير خلال أقل من 200ms بغض النظر عن عدد الموظفين المعروضين.
- **FR-002**: بيانات عرض الموظفين المحسوبة (أيام التواريخ، التنسيقات) MUST تُحسَب مرة واحدة عند تحميل البيانات، وتُعاد الحسبة فقط عند تغيير قائمة الموظفين المُصفَّاة.
- **FR-003**: تغيير نوع التصدير MUST لا يُعيد رسم صفوف أو كروت الموظفين (لأن محتواها لا يتغير بتغيير نوع التصدير).
- **FR-004**: تحديد أو إلغاء تحديد موظف MUST يُعيد رسم الصف أو الكرت المعني فقط، دون المساس بباقي القائمة.
- **FR-005**: عرض الموبايل (الكروت) وعرض الديسكتوب (الجدول) MUST لا يُرسَمان في الـ DOM في نفس الوقت — يُعرض فقط المناسب للشاشة.
- **FR-006**: جميع وظائف التصدير الحالية MUST تظل تعمل بنفس النتائج بعد التحسين (لا تغيير وظيفي).

### Key Entities

- **موظف مع بيانات العرض المحسوبة**: كيان مشتق يحمل (أيام انتهاء العقد، أيام انتهاء الإقامة، أيام التأمين، أيام أجير، والتواريخ المنسَّقة) — يُحسَب مرة واحدة ويُخزَّن.
- **نوع التصدير**: قيمة state بسيطة (`basic` / `monthly`) تؤثر فقط على نموذج الاختيار والـ picker وعملية التصدير الفعلية — لا تؤثر على عرض القائمة.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: تغيير نوع التصدير يستغرق أقل من 200ms (قياساً بأدوات المطور) مع 100+ موظف محمَّل.
- **SC-002**: لا يظهر تجمد مرئي للمتصفح ("jank") عند أي تفاعل في تبويب التصدير.
- **SC-003**: تحديد موظف فردي يستجيب خلال 100ms مع قائمة 200+ موظف.
- **SC-004**: البيانات المُصدَّرة في ملف Excel تطابق البيانات المُصدَّرة قبل التحسين — صفر تغيير في المحتوى.
- **SC-005**: الصفحة تستهلك ذاكرة أقل (لا double DOM rendering).

## Assumptions

- المشكلة موجودة بشكل أساسي عند وجود 100+ موظف (السبب الجذري محدد من التحليل).
- لا يُتوقع تغيير في واجهة المستخدم المرئية — تحسين الأداء فقط خلف الكواليس.
- نفس الـ data fetching المحدودة المرة الواحدة عند تحميل الصفحة تظل كافية.
- الحل لا يحتاج virtualization (قوائم افتراضية) في المرحلة الأولى — `React.memo` + `useMemo` كافيان.
- التحسين يستهدف `ExportTab.tsx` فقط دون تغيير باقي مكونات الاستيراد/التصدير.
